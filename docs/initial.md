# TRPG 在线跑团辅助系统业务设计草案

## 1. 项目定位
项目目标：做一个能够支持基础跑团流程的小型在线房间系统。

核心场景：

1. 用户通过邀请码注册账号。
2. 用户登录后可以创建或加入跑团房间。
3. 房间内成员可以聊天、掷骰、查看成员和角色卡。
4. 系统支持公开掷骰和暗骰。
5. 用户可以创建/管理/修改角色卡。
6. 房间聊天和掷骰记录可以查询和导出。

## 2. 用户角色

系统中存在两类角色关系：系统身份和房间身份。

### 2.1 系统身份

| 身份 | 说明 |
| --- | --- |
| 普通用户 | 通过用户邀请码注册，可以创建房间、加入房间、聊天、掷骰、管理自己的角色卡 |
| 管理员 | 通过管理员邀请码注册，可以管理普通用户、查看用户信息、生成用户邀请码 |

管理员邀请码不由普通用户生成。管理员邀请码可以由系统初始化时预置。

### 2.2 房间身份

| 身份 | 说明 |
| --- | --- |
| 房主 | 创建房间的人，负责管理房间 |
| 主持人 / GM | 跑团主持人，可以辅助管理房间和查看 GM 信息 |
| 玩家 | 房间普通成员，可以聊天、掷骰、使用角色卡 |
| 旁观者 | 只查看房间公开内容，是否能发言由房间规则决定 |
| 系统 Bot | 用于发布掷骰结果和系统提示 |
| Add_on_Bot | 进阶功能，可以提供额外功能的 Bot |

## 3. 功能优先级

### 3.1 基础功能

用户相关：

1. 用户邀请码注册
2. 用户登录
3. 用户退出
4. 用户修改基础个人信息
5. 管理员登录
6. 管理员生成普通用户邀请码

房间相关：

1. 创建房间
2. 生成房间邀请码
3. 使用房间邀请码加入房间
4. 离开房间
5. 查看自己加入的房间
6. 查看房间成员列表
7. 房主关闭房间
8. 房主删除/拉人进入房间

消息相关：

1. 房间普通聊天
2. 保存聊天记录
3. 查看历史消息
4. 区分普通消息、系统消息、掷骰消息

掷骰相关：

1. 普通公开掷骰
2. 带修正值掷骰
3. 暗骰
4. 暗骰结果只对本人可见

角色卡相关：

1. 新建角色卡
2. 修改角色卡
3. 删除角色卡
4. 查看角色卡列表
5. 在房间内选择当前使用的角色卡

日志相关：

1. 查看房间日志
2. 导出自己可见的日志

### 3.2 进阶功能

1. COC 技能检定
2. 使用角色卡技能自动检定
3. 房间日志搜索

5. 管理员查看成员操作日志
6. GM 私密消息
7. 房间公告
8. 房间旁观者模式
9. 指令帮助面板
10. 快捷骰按钮

Add_ons:
1. 头像上传
2. 角色卡导入导出
3. 复杂战斗轮次
4. Bot 插件系统

## 4. 用户管理设计

### 4.1 普通用户

普通用户可以：

1. 使用邀请码注册账号。
2. 登录系统。
3. 退出系统。
4. 修改昵称等基础信息。
5. 创建房间。
6. 加入房间。
7. 管理自己的角色卡。
8. 查看自己的房间和日志。
9. 头像上传。

### 4.2 管理员

管理员可以：

1. 登录系统。
2. 查看成员列表。
3. 查看成员基础信息。
4. 禁用或恢复普通用户。
5. 生成普通用户邀请码。
6. 查看邀请码使用情况。
7. 查看用户操作日志。

管理员不可以查看用户暗骰结果。暗骰结果默认只属于掷骰者本人。

### 4.3 注销账号

1. 管理员可以封禁账号、删除账号

禁用后：

1. 用户不能登录。
2. 用户历史消息保留。
3. 用户角色卡保留但不可继续编辑。
4. 管理员可以查看禁用原因和时间。
5. 管理员可以恢复账号。
6. 管理员可以删除历史消息和角色卡。

## 5. 邀请码设计

邀请码用于控制注册入口。

### 5.1 邀请码类型

| 类型 | 用途 |
| --- | --- |
| 用户邀请码 | 普通用户注册使用 |
| 管理员邀请码 | 管理员注册使用 |

### 5.2 邀请码状态

| 状态 | 说明 |
| --- | --- |
| 未使用 | 可以使用 |
| 已使用 | 已经被使用，不能再次使用 |
| 已过期 | 超过有效期，不能使用 |
| 已禁用 | 管理员手动禁用，不能使用 |

### 5.3 邀请码规则

1. 用户注册必须填写有效邀请码。
2. 管理员注册必须填写管理员邀请码。
3. 普通用户邀请码由管理员生成。
4. 管理员邀请码第一版可以由系统预置。
5. 邀请码生成后需要记录生成者。
6. 邀请码使用后需要记录使用者和使用时间。
7. 邀请码需要有过期时间。
8. 管理员可以禁用邀请码。
## 6. 房间管理设计

### 6.1 房间基础信息

房间应包含：

1. 房间号或房间邀请码。
2. 房间名称。
3. 房主。
4. 成员列表。
5. 创建时间。
6. 房间状态。
7. 房间公告。

### 6.2 房间状态

| 状态 | 说明 |
| --- | --- |
| 开放 | 可以加入、聊天和掷骰 |
| 关闭 | 不再允许新消息，但可以查看历史 |
| 归档 | 只允许房主或管理员查看历史 |
| 删除 | 彻底删除 |
更好的业务规则是关闭或归档房间，这样日志仍然可以查询和导出。
用户可以进一步删除房间，删除所有记录

### 6.3 房间成员

房间成员信息应包含：

1. 用户。
2. 房间身份。
3. 加入时间。
4. 是否离开。
5. 是否被禁言。
6. 当前使用的角色卡。

### 6.4 房间 Bot

每个房间可以有一个系统 Bot。

Bot 的作用：

1. 发布掷骰结果。
2. 发布系统提示。
3. 在零号房间中陪用户测试掷骰。

Bot 不应该像普通用户一样注册和登录，它更像系统中的特殊成员。

## 7. 零号房间设计

零号房间是每个用户的个人空间。

### 7.1 作用

零号房间用于：

1. 测试掷骰。
2. 管理个人角色卡。
3. 查看个人掷骰记录。
4. 导出个人日志。

### 7.2 规则

1. 用户注册成功后自动拥有一个零号房间。
2. 零号房间只有用户本人和系统 Bot。
3. 零号房间不能邀请其他用户加入。
4. 零号房间不能被普通方式关闭。
5. 零号房间日志只能由用户本人查看。

## 8. 消息与日志设计

### 8.1 消息类型

| 类型 | 说明 |
| --- | --- |
| 普通消息 | 用户输入的聊天内容 |
| 掷骰消息 | 系统生成的掷骰结果 |
| 系统消息 | 用户加入房间、离开房间等提示 |
| 指令消息 | 用户输入的指令，可选择是否单独保存 |
| GM 消息 | 只给房主或 GM 查看，进阶功能 |

### 8.2 消息可见性

| 可见性 | 说明 |
| --- | --- |
| 公开 | 房间成员都可以看到 |
| 仅自己可见 | 主要用于暗骰 |
| GM 可见 | 房主和 GM 可见 |
| 指定用户可见 | 后期扩展 |

### 8.3 暗骰规则

暗骰是本项目的重要业务规则。

1. 暗骰结果只对掷骰者本人可见。
2. 其他房间成员不能通过刷新历史消息看到暗骰结果。
3. 管理员默认也不查看用户暗骰结果。
4. 系统可以选择向房间公开一条提示，例如“某用户进行了一次暗骰”，但不能公开结果。

### 8.4 日志导出规则

导出日志时必须遵守消息可见性。

普通玩家导出：

- 公开消息
- 自己的暗骰结果
- 自己可见的系统消息

房主导出：

- 公开消息
- 房间系统消息
- GM 可见消息
- 不包含其他玩家的个人暗骰结果，除非后续业务规则允许

管理员导出：

- 第一版建议不提供管理员导出用户暗骰结果的能力，避免破坏暗骰规则。

## 9. 掷骰系统设计

第一版以 COC 跑团常用需求为主。

### 9.1 基础指令

| 指令 | 说明 |
| --- | --- |
| `.r 1d100` | 公开掷一个 100 面骰 |
| `.r d20` | 公开掷一个 20 面骰 |
| `.r 2d6+3` | 掷两个 6 面骰并加 3 |
| `.rh 1d100` | 暗骰，只给本人看 |

### 9.2 COC 检定

进阶功能可以支持：

| 指令 | 说明 |
| --- | --- |
| `.sc 60` | 手动输入技能值进行 COC 检定 |
| `.check 侦查` | 使用当前角色卡的侦查技能进行检定 |

COC 检定结果可以分为：

1. 大成功
2. 极难成功
3. 困难成功
4. 普通成功
5. 失败
6. 大失败

具体规则可以在后续详细设计中确定。

### 9.3 掷骰记录

掷骰记录需要保存：

1. 掷骰用户。
2. 所在房间。
3. 原始指令。
4. 骰子表达式。
5. 掷骰结果。
6. 是否暗骰。
7. 是否使用角色卡。
8. 使用的角色卡和技能。
9. 创建时间。

## 10. 角色卡设计

### 10.1 基础功能

用户可以：

1. 创建角色卡。
2. 修改角色卡。
3. 删除角色卡。
4. 查看角色卡列表。
5. 查看角色卡详情。
6. 在房间中选择当前使用的角色卡。

### 10.2 角色卡内容

角色卡基础信息：

1. 角色名。
2. 职业。
3. 简介。
4. 所属用户。
5. 创建时间。
6. 更新时间。

角色卡属性示例：

1. 力量
2. 敏捷
3. 意志
4. 体质
5. 外貌
6. 教育
7. 体型
8. 智力

角色卡技能示例：

1. 侦查
2. 聆听
3. 图书馆使用
4. 心理学
5. 斗殴
6. 闪避
7. 幸运

### 10.3 角色卡和房间的关系

角色卡属于用户，但用户可以在不同房间选择不同的当前角色卡。

例如：

1. 用户 A 有三张角色卡。
2. 用户 A 在房间 1 使用角色卡甲。
3. 用户 A 在房间 2 使用角色卡乙。

这样更符合跑团场景。

## 11. 前端业务规划

这里先只规划页面和用户操作，不规划具体前端技术实现。

### 11.1 页面清单

第一版页面：

1. 登录页
2. 注册页
3. 用户主页
4. 房间页
5. 角色卡列表页
6. 角色卡详情页
7. 邀请码管理页
8. 用户管理页
9. 房间日志页

### 11.2 登录页

用户可以：

1. 输入用户名。
2. 输入密码。
3. 登录。
4. 跳转注册页。

### 11.3 注册页

用户可以：

1. 输入用户名。
2. 输入昵称。
3. 输入密码。
4. 确认密码。
5. 输入邀请码。
6. 完成注册。

### 11.4 用户主页

用户主页用于进入主要功能。

需要展示：

1. 我的房间列表。
2. 创建房间入口。
3. 加入房间入口。
4. 零号房间入口。
5. 我的角色卡入口。
6. 当前用户信息。

### 11.5 房间页

房间页是系统核心页面。

需要包含：

1. 房间名称。
2. 房间公告。
3. 成员列表。
4. 当前角色卡摘要。
5. 聊天消息区。
6. 消息输入框。
7. 掷骰指令帮助。
8. 日志入口。
9. 离开房间按钮。

房间页的业务布局可以是：

```text
顶部：房间名称、当前用户、退出/返回
左侧：成员列表、当前角色卡
中间：聊天消息和掷骰结果
右侧：房间公告、指令帮助、快捷操作
底部：消息输入框和发送按钮
```

### 11.6 角色卡页面

角色卡列表页：

1. 查看自己的角色卡。
2. 新建角色卡。
3. 编辑角色卡。
4. 删除角色卡。

角色卡详情页：

1. 编辑基础信息。
2. 编辑属性。
3. 编辑技能。
4. 查看在哪些房间中使用。

### 11.7 管理页面

管理员需要两个主要页面：

1. 用户管理页。
2. 邀请码管理页。

用户管理页：

1. 查看用户列表。
2. 查看用户状态。
3. 禁用或恢复用户。
4. 查看用户注册时间和最后登录时间。

邀请码管理页：

1. 生成用户邀请码。
2. 查看邀请码状态。
3. 查看邀请码使用者。
4. 禁用邀请码。

### 11.8 日志页面

日志页面需要支持：

1. 查看房间历史消息。
2. 搜索关键词。
3. 按消息类型筛选。
4. 导出当前用户可见日志。

## 12. 核心业务流程

### 12.1 注册流程

```text
用户进入注册页
  ↓
填写用户名、昵称、密码、邀请码
  ↓
系统检查邀请码是否有效
  ↓
系统检查用户名是否可用
  ↓
创建用户
  ↓
记录邀请码使用者和使用时间
  ↓
为用户创建零号房间
  ↓
注册完成
```

### 12.2 登录流程

```text
用户进入登录页
  ↓
输入用户名和密码
  ↓
系统检查账号状态
  ↓
系统校验密码
  ↓
登录成功
  ↓
进入用户主页
```

### 12.3 创建房间流程

```text
用户在主页创建房间
  ↓
填写房间名称
  ↓
系统生成房间邀请码
  ↓
创建房间
  ↓
用户成为房主
  ↓
系统 Bot 加入房间
  ↓
进入房间页
```

### 12.4 加入房间流程

```text
用户输入房间邀请码
  ↓
系统检查房间是否存在
  ↓
系统检查房间是否开放
  ↓
系统检查用户是否已经加入
  ↓
用户加入房间
  ↓
进入房间页
```

### 12.5 发送普通消息流程

```text
用户在房间输入普通文本
  ↓
系统检查用户是否为房间成员
  ↓
系统保存消息
  ↓
房间成员可以看到消息
```

### 12.6 公开掷骰流程

```text
用户输入公开掷骰指令
  ↓
系统识别指令
  ↓
系统生成随机结果
  ↓
系统保存掷骰消息
  ↓
房间成员都可以看到结果
```

### 12.7 暗骰流程

```text
用户输入暗骰指令
  ↓
系统识别指令
  ↓
系统生成随机结果
  ↓
系统保存暗骰结果
  ↓
只有掷骰者本人可以看到结果
  ↓
其他成员最多看到一次暗骰提示
```

### 12.8 角色卡技能检定流程

```text
用户在房间选择当前角色卡
  ↓
用户输入技能检定指令
  ↓
系统查找当前角色卡对应技能
  ↓
系统进行 1d100 检定
  ↓
系统判断成功等级
  ↓
系统保存并展示检定结果
```
